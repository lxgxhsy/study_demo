# Spec Delta: Leaf ID

## ADDED Requirements

### Requirement: Segment-Based ID Generation

The system shall generate IDs per business tag using database-backed segment allocation.

#### Scenario: Generate ID for business tag

- **WHEN** a client requests `GET /api/lab/id/{bizTag}`
- **THEN** the system returns an ID from the current in-memory segment
- **AND** the response includes `bizTag`, `id`, and `requestId`

#### Scenario: Single-instance monotonicity

- **WHEN** one service instance generates multiple IDs for the same `bizTag`
- **THEN** returned IDs are monotonically increasing for that instance

#### Scenario: Allocate new segment

- **WHEN** the current segment is exhausted
- **THEN** the system allocates a new segment from the database
- **AND** the new segment range does not overlap with previously allocated ranges

#### Scenario: H2 validation boundary

- **WHEN** tests run under the `local-h2` profile
- **THEN** the result may validate functional ID generation flow
- **AND** the result must not be presented as proof of MySQL/InnoDB segment concurrency safety

#### Scenario: MySQL concurrency validation boundary

- **WHEN** tests run under the `mysql-leaf` profile
- **THEN** the test may be used as segment allocation concurrency evidence
- **AND** the allocation strategy uses optimistic update with `WHERE version = ?` and retry, or a transactional row-lock strategy

### Requirement: Concurrent Duplicate Protection

The system shall prevent duplicate IDs under concurrent requests.

#### Scenario: Concurrent batch generation across segments

- **WHEN** concurrent generation runs for one `bizTag` with concurrency 64
- **AND** total generated IDs are at least `step * 3`
- **THEN** generated IDs cross the current segment, the preloaded segment, and a newly allocated DB segment
- **AND** `duplicateCount=0`
- **AND** allocated segment ranges do not overlap

#### Scenario: Batch size limit

- **WHEN** a client submits `POST /api/lab/id/{bizTag}/batch` with a count greater than the documented maximum
- **THEN** the system returns HTTP 400
- **AND** the error code is `ID_BATCH_SIZE_TOO_LARGE`

### Requirement: Double Buffer Preload

The system shall preload the next segment before the current segment is exhausted.

#### Scenario: Preload below threshold

- **WHEN** remaining IDs in the current segment fall below the configured threshold
- **THEN** the system starts asynchronous loading for the next segment
- **AND** the default threshold is 10% of `step`

#### Scenario: Timeout while waiting for async preload

- **WHEN** the current segment is exhausted
- **AND** async preload is still running
- **THEN** request threads wait no longer than `preload-wait-timeout-ms`
- **AND** timeout returns HTTP 500 with error code `ID_SEGMENT_ALLOC_FAILED`

#### Scenario: Switch segment

- **WHEN** the current segment is exhausted and the next segment is ready
- **THEN** the system switches to the next segment without allocating duplicate IDs

#### Scenario: ID metrics

- **WHEN** a client requests `GET /api/lab/id/{bizTag}/metrics`
- **THEN** the response includes `currentStart`, `currentEnd`, `currentValue`, `remaining`, `nextReady`, `preloadRunning`, `dbAllocationCount`, and `segmentSwitchCount`

### Requirement: Order Lab ID Use

The system shall use the Leaf ID generator in a minimal high-concurrency order-lab endpoint after the ID module is independently verified.

#### Scenario: Create lab order

- **WHEN** a client submits `POST /api/lab/orders`
- **THEN** the response includes an `orderId` generated by the Leaf ID generator
- **AND** the response includes async accepted, rejected, and caller-runs task counts
- **AND** the endpoint does not implement payment, user accounts, inventory persistence, or real downstream integration
